## Why

The diagnostic evidence collected for issue [#218](https://github.com/kt-npm-modules/svelte-adapter-azure-swa/issues/218) — captured by the previously archived diagnostics changes [`2026-05-29-forwarded-headers-diagnostics`](../archive/2026-05-29-forwarded-headers-diagnostics/) and [`2026-05-29-diagnose-swa-rewrite-vs-fallback`](../archive/2026-05-29-diagnose-swa-rewrite-vs-fallback/) — demonstrates that real Azure SWA routinely **overwrites or injects an `Authorization` header** before the request reaches the managed Function, on every adapter-supported HTTP method (`GET`, `HEAD`, `POST`, `PUT`, `PATCH`, `DELETE`, `OPTIONS`) and through both `navigationFallback` and explicit per-path `rewrite` routing. The injection happens even when the client never sent an `Authorization` header. The local SWA CLI emulator does not reproduce this behaviour. Because SvelteKit endpoints, hooks, and SSR loads currently observe whatever `Authorization` value Azure SWA chose to set, every app deployed through this adapter is silently exposed to a platform-injected bearer token in `request.headers.get('authorization')` — typically interpreted by user code as the end-user's credential.

The minimal correct adapter behaviour is to stop forwarding the `Authorization` header to SvelteKit by default, while preserving an explicit opt-in for users who genuinely want the platform-supplied value. This change implements that policy, refactors the existing `x-adapter-test-workarounds` JSON payload into a typed namespaced shape so the new auth diagnostics fit alongside the existing empty-form workaround diagnostics, and updates the demo's diagnostic-headers e2e expectations to assert the fixed default behaviour against both the local SWA CLI emulator and a real Azure SWA deployment.

## What Changes

- **Add adapter option `preserveAuthorization?: boolean`** to `Options` in [src/index.d.ts](src/index.d.ts), defaulting to `false`. When `false` or omitted, the adapter SHALL strip the inbound `Authorization` header (case-insensitively) before constructing the SvelteKit `Request`. When `true`, the adapter SHALL preserve the current behaviour and forward `Authorization` unchanged.
- **Thread `preserveAuthorization` through the generated server entry**: extend [src/server/index.js](src/server/index.js) `writeEnvironment` to emit `preserveAuthorization` into the generated `ENV` module in the same style as `debug` and `testWorkarounds`. Extend the `ENV` module typing in [src/server/entry/index.d.ts](src/server/entry/index.d.ts) to declare the new boolean export. Import the flag in [src/server/entry/entry.js](src/server/entry/entry.js).
- **Strip `Authorization` in the request-construction path** of [src/server/entry/entry.js](src/server/entry/entry.js): when `preserveAuthorization` is `false`, the header iteration that copies `httpRequest.headers` into the downstream `Request` headers SHALL skip any header whose name lowercases to `authorization`. The existing `x-ms-original-url` filter and the `Request.url` construction from `x-ms-original-url` are unchanged.
- **Refactor `testWorkaroundsInfo` into a typed nested shape** under the existing single transport header `x-adapter-test-workarounds`. `x-adapter-test-workarounds` remains the only test-workaround transport header — no new sibling headers are introduced; only its JSON payload is refactored into namespaces. Replace the loose `Record<string, any>` typedef with a JSDoc `AdapterTestWorkaroundsInfo` typedef containing two optional namespaces: `emptyFormContentTypeStrip` (carrying the existing empty-form fields) and `auth` (carrying the new auth-workaround fields). No mixing of namespaces at the same object level.
- **Add auth workaround diagnostics** under `testWorkaroundsInfo.auth` when `testWorkarounds` is `true`. The fields are `rawAuthorizationPresent: boolean`, `testWorkaroundAuthorizationPresent: boolean`, `rawAuthorizationEqualsTestWorkaroundAuthorization: boolean | null`, and `authorizationStripped: boolean`. The presence/equality facts are computed from the inbound `Authorization` and the new test control header `x-test-workaround-authorization` BEFORE the strip step. Raw header values, including Azure-injected bearer tokens, are NEVER copied into the JSON payload.
- **Add adapter unit coverage** in `tests/unit/` for the request/header copy behaviour. The preferred path is to extract a small internal helper next to `entry.js` (e.g. `src/server/entry/copy-headers.js` or similarly named) covering the deterministic header-copy / strip / auth-workaround-info computation, and unit-test that helper directly. Exporting `toRequest` from `entry.js` is the fallback if helper extraction proves clearly worse after inspecting the current code. No public API surface is added beyond `preserveAuthorization`.
- **Update the existing `tests/demo/e2e/diagnostic-headers.test.ts` expectations** for the fixed default behaviour. After this change, `Authorization` is absent from SvelteKit-level diagnostic facts on both the local SWA CLI and real Azure SWA; `authorizationEqualsTestAuthorization` is `null` because `Authorization` was stripped. The adapter's `x-adapter-test-workarounds` auth-namespace facts are asserted against the four matrix cells (local-with-auth, local-no-auth-baseline, Azure-with-auth, Azure-no-auth-baseline) using the same `PUBLIC_SWA_CLI` / `CI` environment pattern the existing empty-form test already uses.
- **Document `preserveAuthorization`** in the README options section, alongside the existing `debug` and `testWorkarounds` entries. The doc explains the default-off rationale (Azure SWA injects/overwrites `Authorization` on managed Functions), the opt-in semantics for the escape hatch, and a caveat that apps relying on a client-supplied bearer token behind SWA likely need an app-specific custom header rather than `Authorization`.
- **BREAKING (default behaviour change)**: SvelteKit code in adapter consumers will, by default, no longer observe an `Authorization` request header — neither a value the client sent, nor a value Azure SWA injected. Consumers who need the previous behaviour set `preserveAuthorization: true`. Consumers who relied on `Authorization` for application bearer-auth on an Azure-deployed app should evaluate whether their reliance was already broken by Azure's injection (likely yes) and switch to an app-specific custom header.

## Capabilities

### New Capabilities

- `adapter-authorization-policy`: the adapter's policy and request-mutation contract for the inbound `Authorization` header — the new `preserveAuthorization` option, the strip-by-default behaviour, the case-insensitive matching rule, the unchanged `x-ms-original-url` consumption, and the typed nested `x-adapter-test-workarounds` JSON contract (`emptyFormContentTypeStrip` + `auth` namespaces) that lets tests observe the pre-strip Authorization comparison without exposing raw secrets.

### Modified Capabilities

- `demo-diagnostics`: tighten the diagnostic-headers expectations so that, after the adapter fix, SvelteKit-level diagnostic facts no longer report `Authorization` presence by default on either local SWA CLI or real Azure SWA, and the four matrix cells of pre-strip auth-workaround facts are asserted against `x-adapter-test-workarounds.auth`. This preserves the diagnostic surface as a regression check for the policy.

- **Validation**: `npm run format`, `npm run lint`, `npm run check`, `npm run test:swa --prefix tests/demo`, `openspec validate strip-swa-authorization --strict`.
